Vue Setup Ref() v.s. reactive()​

url:

  1. https://medium.com/@bsalwiczek/ref-vs-reactive-in-vue-3-whats-the-right-choice-7c6f7265ce39
  2. https://medium.com/i-am-mike/vue-3-ref-%E8%B7%9F-reactive-%E6%88%91%E8%A9%B2%E6%80%8E%E9%BA%BC%E9%81%B8-2fb6b6735a3c
  3. https://medium.com/swlh/vue3-using-ref-or-reactive-88d47c8f6944
  4. https://www.readfog.com/a/1633537209551392768

Quick introduction​

Ref() 和 reactive() 都是用來定義好要用來追蹤的參數。 使用這它們就是要告訴 Vue 說 如果用這兩個(Ref(), reactive())定義好的參數有發生任何改變，要 re-build 或 re-evaluate everything that depends on those variables.

Difference​

ref : 可以接受任何型態的資料，但是不會對物件或陣列內部的屬性變動做監聽，取資料一定要 .value.

refs are unwrapped when they are passed to templates, that’s why you don’t need to write .value there.

reactive : 只能接受 Object 或 Array，會對內部的屬性變動做深層的監聽，取資料時不需要 .value。

# 官方文件
The reactive conversion is "deep": it affects all nested properties. A reactive object also deeply unwraps any properties that are refs while maintaining reactivity.

Theory​

其實去看 Vue 在實作 ref 的時候，發現其實底層也是用 reactive 實作出來的。

class RefImpl<T> {
  private _value: T
  private _rawValue: T

  public dep?: Dep = undefined
  public readonly __v_isRef = true

  constructor(value: T, public readonly __v_isShallow: boolean) {
    this._rawValue = __v_isShallow ? value : toRaw(value)
    this._value = __v_isShallow ? value : toReactive(value)
  }

  get value() {
    trackRefValue(this)
    return this._value
  }

  set value(newVal) {
    newVal = this.__v_isShallow ? newVal : toRaw(newVal)
    if (hasChanged(newVal, this._rawValue)) {
      this._rawValue = newVal
      this._value = this.__v_isShallow ? newVal : toReactive(newVal)
      triggerRefValue(this, newVal)
    }
  }
}

當我們寫 ref(x) 其實去觸發 constructor，這時候發現 constructor 其實裡面是 reactive. 所以我們可以想像成 ref 其實是：

const myRef = reactive({
  value: "I'm ref!"
})

myRef.value // "I'm ref!"
myRef.value = "Changed"
myRef.value // "Changed"

Why const ref is valid?​

我們舉個例子來說：

import { ref } from "vue";

export default {
  setup() {
    const title = ref("Hello, Vue 3!");

    return { title };
  }
};

如果我們把 title console 出來就會發現：

{_isRef: true}
value: (...)
_isRef: true
get value: ƒ value()
set value: ƒ value(newVal)
__proto__: Object

我們改的是 ref.value 而不是直接去改到 ref.

Conclusion​

  • 大部分時候其實還是會選擇用 ref 居多，只有在很多資料需要統整或是要對 objet 內部的資料去做監控的時候就會選擇用 reactive。
  • ref()定義的響應式數據需要通過.value 來訪問，而在模板中會進行一個拆箱的操作，不需要手動通過.value 來訪問。reactive()函數返回的對象需要在模板裏通過.操作符訪問。
  • ref()可以爲基本類型和引用類型值創建響應式數據，而爲引用類型創建響應式數據時，內部還是調用了 reactive()。而 reactive()只能接收一個對象，我們可以把一些相關聯的數據都放在這個對象裏，可以提高代碼的可讀性。
  • 如果邏輯可以複用可以使用組合式函數，這樣其他組件也可以使用這個邏輯。reactive()函數返回的對象如果被解構的話，裏面的數據將會失去響應式，可以通過 toRefs 把對象裏面的每個屬性轉化成 ref 來使用。
